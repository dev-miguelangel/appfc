AppFC

Documentación técnica — estado actual del proyecto

AppFC es una plataforma para coordinar partidos de fútbol amateur en Chile. Permite a jugadores crear y gestionar equipos, programar partidos con confirmación bilateral, registrar resultados y mantener reputación basada en asistencia y compromiso.

Modo local (Mock)

Cuando environment.useLocalDb = true, el token SUPABASE_CLIENT provee un MockSupabaseClient que simula la misma API superficial de @supabase/supabase-js usando localStorage como base de datos. Esto permite desarrollar sin conexión a Supabase.

El LocalStore persiste arrays JSON bajo claves ldb_<tabla>. El QueryBuilder del mock soporta: select(), eq(), neq(), ilike(), order(), limit(), insert(), update(), delete(), single(), upsert().

07 Reglas de Negocio

Equipos

⚽
Solo capitanes crean partidos
En el formulario Nuevo Partido, el equipo local solo muestra equipos donde capitan_id === userId. Un jugador sin equipos capitaneados no puede crear partidos.
🏃
Un equipo no puede jugar contra sí mismo
Reforzado en BD con CHECK (equipo_local_id <> equipo_visitante_id) y en el selector del formulario que excluye el equipo ya elegido como local.

Partidos — flujo de estados

  CREACIÓN
      ↓ (capitán local crea, visitante = pendiente)
  programado  ←──────────────────────────────────────┐
      │                                                   │
      │ cap. visitante rechaza                            │ (futuro: mediación)
      ▼                                                   │
  programado (aceptacion_visitante = rechazada)         │
      │                                                   │
      │ cap. visitante acepta                             │
      ▼                                                   │
  programado (aceptacion_visitante = aceptada)          │
      │                                                   │
      ├──── cap. local cancela ──────────► cancelado        │
      │                                                   │
      │ ambos capitanes confirman marcador igual          │
      ▼                                                   │
  completado  (reputación calculada por trigger)         │
      │                                                   │
      │ ambos capitanes confirman marcador diferente      │
      ▼                                                   │
  en_disputa  ────── revisión manual ──────────────────┘

Convocatoria automática

📋
Auto-convocatoria al crear partido
Al crear un partido, PartidosService.autoConvocar() itera sobre los miembros activos de ambos equipos y crea registros en partido_jugadores con estado: 'pendiente'. Evita duplicados con un Set<userId>.
✅
Respuesta individual de convocatoria
Cada jugador puede confirmar o rechazar su convocatoria desde la vista de detalle del partido. Solo visible si estado_convocatoria === 'pendiente'.

Asistencia

🔒
Capitán solo pasa asistencia de su equipo
El computed miEquipoComoCapitan identifica si el usuario es capitán del equipo local o visitante. Los checkboxes de asistencia solo se muestran para el equipo propio. En el servicio, la consulta incluye .eq('equipo_id', equipo_id).

Resultado bilateral y solicitudes

🤝
El capitán visitante debe aceptar el partido
Al crear un partido, aceptacion_visitante = 'pendiente'. El capitán visitante ve un banner de aceptación/rechazo. Hasta que acepte, no puede solicitar cambios.
📝
Visitante propone, local aprueba
El capitán visitante (si aceptó) puede solicitar edición de fecha/lugar o cancelación. La solicitud queda en estado: 'pendiente' hasta que el local la apruebe o rechace. Solo puede haber una solicitud pendiente por visitante.
⚖️
Confirmación bilateral de marcador
Cada capitán ingresa el marcador completo (goles_local, goles_visitante). En Supabase, el trigger trg_verificar_resultado compara los dos JSONb. En modo mock, ResultadosService.resolverBilateralMock() hace lo mismo en el cliente.

08 Sistema de Reputación

La reputación se calcula automáticamente cuando un partido queda completado. Los tres índices están clampeados en [0, 100] con default inicial de 50.

Condiciónrep_asistenciarep_puntualidadrep_compromiso
confirmado + asistió+3+2+2
confirmado + NO asistió−6−4−3
pendiente + asistió+1——
rechazado (cualquiera)———
ℹ️

En producción, el cálculo lo ejecuta la función PostgreSQL public.recalcular_reputacion_partido(p_partido_id uuid). En modo mock, ResultadosService.actualizarReputacionesMock() replica la misma lógica en el cliente y llama a auth.loadPerfil(uid) para refrescar el signal reactivo.

10 Feature Flags

Los flags se definen en src/environments/environment.ts (dev) y environment.prod.ts (prod).

environment.useLocalDb boolean
true → Inyecta MockSupabaseClient. Sin conexión de red. Datos en localStorage bajo prefijo ldb_.
false → Usa createClient(supabaseUrl, supabaseAnonKey) real.
⚠ Siempre false en producción.
environment.production boolean
Controla el Service Worker a través de provideServiceWorker('ngsw-worker.js', { enabled: !isDevMode() }).
También usado en AuthComponent.isDev para mostrar el formulario email/contraseña.
environment.supabaseUrl string
URL del proyecto Supabase. Ej: https://xxxx.supabase.co. Ignorado cuando useLocalDb = true.
environment.supabaseAnonKey string
Clave anónima pública del proyecto Supabase. No es un secreto — es la clave de acceso público protegida por RLS.

Configuración actual de desarrollo

export const environment = {
  production:   false,
  useLocalDb:   true,   // ← cambiar a false para probar con Supabase real
  supabaseUrl:  'YOUR_SUPABASE_URL',
  supabaseAnonKey: 'YOUR_SUPABASE_ANON_KEY',
};

3. Configuración de Storage (bucket fotos)

El bucket debe tener una política de lectura pública. Ejecutar en el SQL Editor:

INSERT INTO storage.buckets (id, name, public)
VALUES ('fotos', 'fotos', true)
ON CONFLICT (id) DO NOTHING;

CREATE POLICY "Public read fotos"
  ON storage.objects FOR SELECT USING (bucket_id = 'fotos');

CREATE POLICY "Auth upload fotos"
  ON storage.objects FOR INSERT TO authenticated
  WITH CHECK (bucket_id = 'fotos');

4. Verificar trigger de signup

Confirmar que el trigger on_auth_user_created existe. Si no, ejecutar:

CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  INSERT INTO public.usuarios (id, nombre, rep_asistencia, rep_puntualidad, rep_compromiso, created_at)
  VALUES (NEW.id, '', 50, 50, 50, now());
  RETURN NEW;
END;
$$;

CREATE OR REPLACE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

13 Guía Supabase — Paso a Paso

Crear proyecto

  1. Ingresar a supabase.com → New Project.
  2. Elegir organización, nombre del proyecto (appfc), contraseña de BD y región más cercana a Chile (São Paulo o US East).
  3. Esperar ~2 minutos a que el proyecto se inicialice.
  4. Ir a Project Settings → API y copiar:
    • Project URL → environment.supabaseUrl
    • anon public key → environment.supabaseAnonKey

Ejecutar migraciones

  1. En Supabase, ir a SQL Editor → New query.
  2. Copiar y ejecutar el contenido de 20260407_etapa2_usuarios.sql. Verificar que no hay errores.
  3. Repetir para cada migración en orden cronológico. No saltar ninguna: cada migración asume que las anteriores ya existen.
  4. Verificar en Table Editor que las 8 tablas aparecen: usuarios, equipos, equipo_miembros, partidos, partido_jugadores, partido_solicitudes, notificaciones.
  5. Verificar en cada tabla que Row Level Security está habilitado (ícono de candado).

Configurar Google OAuth

  1. Ir a console.cloud.google.com → APIs & Services → OAuth consent screen → Crear app (tipo Externo).
  2. Ir a Credentials → Create Credentials → OAuth Client ID → Tipo: Web Application.
  3. Agregar en Authorized JavaScript origins: https://TU-PROYECTO.supabase.co y tu dominio de producción.
  4. Agregar en Authorized redirect URIs: https://TU-PROYECTO.supabase.co/auth/v1/callback.
  5. Copiar el Client ID y Client Secret.
  6. En Supabase → Authentication → Providers → Google → pegar Client ID y Client Secret → Save.
  7. En Supabase → Authentication → URL Configuration → agregar a Redirect URLs: https://TU-DOMINIO.com/auth/callback.

Configurar Storage

  1. Ir a Storage → New Bucket → nombre: fotos → marcar Public bucket → Create bucket.
  2. En el bucket fotos, ir a Policies → New Policy.
  3. Crear política SELECT para Everyone (lectura pública de fotos).
  4. Crear política INSERT para Authenticated users (solo usuarios logueados pueden subir).

Activar modo Supabase en desarrollo

  1. Editar src/environments/environment.ts: 
      useLocalDb: false,
  supabaseUrl: 'https://xxxx.supabase.co',
  supabaseAnonKey: 'eyJ...',
  2. Ejecutar ng serve. La app ahora habla con Supabase real.
  3. Registrarse con email/contraseña (la primera cuenta deberá confirmar email, a menos que deshabilites email confirmation en Authentication → Settings).
  4. Para deshabilitar confirmación de email en desarrollo: Supabase → Authentication → Settings → Enable email confirmations → OFF.

Usar Supabase CLI (opcional, para migraciones automáticas)

  1. Instalar CLI: npm install -g supabase
  2. Login: supabase login
  3. Inicializar en el proyecto: supabase init (ya existe supabase/)
  4. Enlazar con el proyecto remoto: supabase link --project-ref TU_PROJECT_REF
  5. Subir migraciones: supabase db push
  6. Ver estado: supabase db remote changes
