01 · Resumen ejecutivo

Vision tecnica

AppFC es una SPA Angular servida en Vercel. Todo el backend vive en Supabase: base de datos PostgreSQL, autenticacion OAuth, storage para avatares y canales realtime para notificaciones en vivo. No se requiere servidor propio para el MVP.

Principio de diseno
Cero servidores propios en v1. Supabase actua como BaaS completo. Angular consume la API de Supabase directamente desde el cliente usando Row Level Security para garantizar que cada usuario solo accede a sus propios datos.

Lo que resuelve esta arquitectura

→
Sin servidor intermedio
Angular llama directo a Supabase. Menos latencia, menos infra, menos costo en fase inicial.
→
Seguridad por defecto
Row Level Security en PostgreSQL garantiza aislamiento de datos sin logica adicional en el frontend.
→
Escalable cuando sea necesario
Se puede agregar Edge Functions de Supabase para logica de negocio compleja sin redisenar la arquitectura.

Diagrama entidad-relacion

PostgreSQL 17 en Supabase. Las tablas auth.users son gestionadas por Supabase Auth. El resto vive en el schema public.

Indices y constraints clave

SQL supabase/migrations/001_schema.sql 
-- ENUMs
CREATE TYPE posicion_enum    AS ENUM ('portero', 'defensa', 'volante', 'delantero');
CREATE TYPE tipo_futbol_enum  AS ENUM ('futbol_11', 'futbol_7', 'futbol_5', 'futsal');
CREATE TYPE estado_partido    AS ENUM ('programado', 'en_disputa', 'completado', 'cancelado');
CREATE TYPE invitacion_enum   AS ENUM ('pendiente', 'aceptado', 'rechazado');
CREATE TYPE superficie_enum   AS ENUM ('cesped_natural', 'cesped_sintetico', 'cemento', 'parquet');

-- Tabla lugares (canchas)
CREATE TABLE lugares (
  id               uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  nombre           varchar(100) NOT NULL,
  direccion        text,
  comuna           varchar(60),
  tipo_superficie  superficie_enum,
  precio_hora      integer,
  lat              numeric(9,6),
  lng              numeric(9,6),
  foto_url         text,
  contacto         varchar(80),
  es_patrocinador  boolean      NOT NULL DEFAULT false,
  created_at       timestamptz  NOT NULL DEFAULT now()
);

-- Tabla partidos (actualizada)
CREATE TABLE partidos (
  id                   uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  titulo               varchar(100),
  tipo_futbol          tipo_futbol_enum NOT NULL,
  max_jugadores_equipo smallint    NOT NULL,
  fecha_hora           timestamptz NOT NULL,
  lugar_id             uuid        REFERENCES lugares(id),
  equipo_local_id      uuid        NOT NULL REFERENCES equipos(id),
  equipo_visitante_id  uuid        NOT NULL REFERENCES equipos(id),
  goles_local          smallint,   -- NULL hasta cerrar el partido
  goles_visitante      smallint,
  result_conf_local    boolean     NOT NULL DEFAULT false,
  result_conf_visit    boolean     NOT NULL DEFAULT false,
  estado               estado_partido NOT NULL DEFAULT 'programado',
  created_at           timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT equipos_distintos CHECK (equipo_local_id != equipo_visitante_id)
);

-- partido_jugadores ahora lleva equipo_id
ALTER TABLE partido_jugadores
  ADD COLUMN equipo_id uuid NOT NULL REFERENCES equipos(id);

-- Estado pasa a completado solo cuando AMBOS capitanes confirman
CREATE OR REPLACE FUNCTION check_resultado_confirmado()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.result_conf_local AND NEW.result_conf_visit THEN
    NEW.estado := 'completado';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_resultado_confirmado
  BEFORE UPDATE ON partidos
  FOR EACH ROW EXECUTE FUNCTION check_resultado_confirmado();

-- Indices clave
CREATE INDEX idx_usuarios_comuna_posicion   ON usuarios (comuna, posicion);
CREATE INDEX idx_partidos_fecha             ON partidos (fecha_hora);
CREATE INDEX idx_partidos_equipos           ON partidos (equipo_local_id, equipo_visitante_id);
CREATE INDEX idx_partido_jugadores_partido  ON partido_jugadores (partido_id);
CREATE INDEX idx_notif_usuario_leida        ON notificaciones (usuario_id, leida);
CREATE INDEX idx_lugares_comuna             ON lugares (comuna);
CREATE INDEX idx_lugares_patrocinador       ON lugares (es_patrocinador) WHERE es_patrocinador = true;

-- Funcion que recalcula reputacion tras registrar asistencia
CREATE OR REPLACE FUNCTION recalcular_reputacion(p_usuario_id uuid)
RETURNS void AS $$
  UPDATE usuarios SET
    rep_asistencia = (
      SELECT ROUND(AVG(asistio::int) * 100, 1)
      FROM  partido_jugadores
      WHERE usuario_id = p_usuario_id AND asistio IS NOT NULL
    )
  WHERE id = p_usuario_id;
$$ LANGUAGE sql;

05 · Row Level Security

Politicas de seguridad

RLS es la capa de autorizacion principal. Cada tabla tiene politicas que se evaluan en PostgreSQL antes de retornar datos al cliente Angular.

TablaOperacionPoliticaDescripcion
usuarios SELECT auth.uid() = id Cada usuario lee solo su propio perfil. Para ver otro jugador existe una funcion RPC publica.
usuarios UPDATE auth.uid() = id Solo el propio usuario puede editar su perfil.
equipos SELECT Es miembro activo del equipo Solo miembros ven el equipo.
equipos INSERT Usuario autenticado Cualquier usuario logueado puede crear un equipo.
equipos UPDATE / DELETE capitan_id = auth.uid() Solo el capitan puede modificar o eliminar el equipo.
partidos SELECT Es convocado o capitan Solo jugadores invitados o el capitan ven el partido.
partidos INSERT Usuario es capitan de ambos equipos o de uno con invitacion al otro El partido debe referenciar dos equipos distintos.
partidos UPDATE resultado Es capitan del equipo local o visitante segun el campo a confirmar Cada capitan solo puede confirmar el resultado de su equipo (result_conf_local o result_conf_visit).
partido_jugadores UPDATE asistio Capitan del equipo del jugador en ese partido Cada capitan registra asistencia solo de los jugadores de su equipo.
notificaciones SELECT / UPDATE usuario_id = auth.uid() Cada usuario solo ve y marca sus propias notificaciones.
lugares SELECT Publica (cualquier usuario autenticado) Cualquier usuario puede ver el directorio de canchas al crear un partido.
lugares INSERT / UPDATE Solo via service role (admin) Los recintos no se auto-registran en el MVP; lo hace un admin via Supabase Studio o panel interno.
06 · Autenticacion

Auth con Angular + Supabase

TypeScript auth.service.ts 
import { Injectable, inject, signal, computed } from '@angular/core';
import { SupabaseClient, Session } from '@supabase/supabase-js';

@Injectable({ providedIn: 'root' })
export class AuthService {
  private supabase = inject(SUPABASE_CLIENT);

  // Estado reactivo con Signals
  readonly session = signal<Session | null>(null);
  readonly isLoggedIn = computed(() => this.session() !== null);
  readonly userId    = computed(() => this.session()?.user.id);

  constructor() {
    // Inicializar desde sesion activa
    this.supabase.auth.getSession().then(({ data }) =>
      this.session.set(data.session)
    );
    // Escuchar cambios de sesion
    this.supabase.auth.onAuthStateChange((_, s) =>
      this.session.set(s)
    );
  }

  loginWithGoogle() {
    return this.supabase.auth.signInWithOAuth({
      provider: 'google',
      options: { redirectTo: `${location.origin}/app/dashboard` }
    });
  }

  logout() { return this.supabase.auth.signOut(); }
}

// Guard funcional Angular 20
export const authGuard: CanActivateFn = () => {
  const auth   = inject(AuthService);
  const router = inject(Router);
  return auth.isLoggedIn() || router.createUrlTree(['/auth']);
};
07 · Realtime

Notificaciones en vivo

Supabase Realtime usa Change Data Capture sobre PostgreSQL. Angular se suscribe via WebSocket y actualiza Signals sin polling.

TypeScript notificaciones.service.ts 
export class NotificacionesService {
  private supabase = inject(SUPABASE_CLIENT);
  private auth     = inject(AuthService);

  readonly notifs = signal<Notificacion[]>([]);
  readonly unread = computed(() => this.notifs().filter(n => !n.leida).length);

  subscribeToRealtime() {
    this.supabase
      .channel(`notifs-${this.auth.userId()}`)
      .on('postgres_changes', {
        event:  'INSERT',
        schema: 'public',
        table:  'notificaciones',
        filter: `usuario_id=eq.${this.auth.userId()}`
      }, (payload) => {
        this.notifs.update(n => [payload.new as Notificacion, ...n]);
      })
      .subscribe();
  }
}
08 · Reglas de negocio

Logica central de la aplicacion

Implementadas como combinacion de RLS policies, funciones PostgreSQL y validaciones en el servicio Angular segun complejidad.

Equipos y membresia

01
Capitan unico por equipo
El creador del equipo es automaticamente su capitan. Solo puede existir un capitan activo. Transferencia de capitan es posible pero manual.
02
Pertenencia multiple
Un usuario puede ser miembro de multiples equipos sin restriccion. El constraint UNIQUE aplica sobre (equipo_id, usuario_id).
03
Invitacion obligatoria
No existe union libre a un equipo. Siempre requiere invitacion del capitan. El estado inicial es pendiente.

Partidos

04
Obligatorio: 2 equipos distintos
Todo partido requiere equipo_local_id y equipo_visitante_id NOT NULL y distintos. Garantizado por CHECK constraint en la DB.
05
Tipo de futbol y cupo maximo
tipo_futbol (futbol_11 / futbol_7 / futbol_5 / futsal) y max_jugadores_equipo son campos requeridos al crear el partido. La app valida que no se convoquen mas jugadores que el cupo por equipo.
06
Solo el capitan convoca
Ningun miembro puede crear partidos sin ser capitan del equipo. Cada capitan gestiona la convocatoria de su propio equipo.

Resultado del partido

07
Confirmacion bilateral obligatoria
Cada capitan ingresa los goles de su equipo y confirma el resultado (result_conf_local / result_conf_visit). El partido pasa a completado solo cuando AMBOS confirman, via trigger PostgreSQL.
08
Resultado en disputa
Si ambos capitanes confirman pero los marcadores no coinciden, el partido queda en estado en_disputa y se notifica a ambos para corregir. La reputacion no se actualiza hasta resolver.
09
Ventana de confirmacion
Los capitanes tienen 24 horas despues de la fecha del partido para confirmar el resultado. Pasado ese plazo, un Edge Function cancela automaticamente la confirmacion pendiente y envia recordatorio.

Lugares y canchas

10
Lugar referenciado, no texto libre
Al crear un partido, el lugar se selecciona del catalogo de la tabla lugares via FK. Si no existe la cancha, el capitan puede sugerirla (flujo separado de alta manual por admin).
11
Visibilidad de canchas patrocinadoras
En el selector de lugar, las canchas con es_patrocinador = true aparecen primero con un badge destacado. El orden es: patrocinadores por comuna → resto por comuna → otras comunas.

Reputacion

14
Calculo post-partido
La reputacion se recalcula via funcion PostgreSQL una vez que el partido esta completado (ambos capitanes confirmaron). Es un promedio historico ponderado.
15
Visibilidad controlada
El puntaje de reputacion es visible publicamente en la busqueda de reemplazos. El historial detallado es privado.

Busqueda de reemplazos

12
Filtro por posicion y comuna
La query de candidatos filtra por: misma posicion que el jugador baja, misma comuna, no convocado ya al partido, ordenado por reputacion desc.
13
Limite de invitaciones
Se pueden enviar hasta 5 invitaciones simultaneas por cupo vacante para no saturar a los jugadores libres.

Estrategia de base de datos

→
Connection pooling desde dia 1
Configurar el cliente Angular con el endpoint de PgBouncer de Supabase (puerto 6543) para no agotar conexiones.
→
Particion por fecha en partidos
Cuando la tabla partidos supere 1M de filas, particionar por ano/mes para mantener queries rapidas.
→
Full-text search con pg_trgm
Para busqueda de jugadores por nombre, activar pg_trgm y crear indice GIN en la columna nombre.

Estrategia de frontend

→
Lazy loading agresivo
Cada modulo Angular se carga solo cuando el usuario navega a esa ruta. El bundle inicial debe ser menor a 150kb.
→
SSR con Angular Universal (v2)
Activar SSR en Vercel para mejorar SEO de las paginas publicas (landing, perfiles publicos de jugadores).
→
PWA para movil
Service Worker + Web Push Notifications para comportamiento nativo sin app store. Instalar en pantalla de inicio.
